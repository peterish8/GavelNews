<objective>
Replace the plain flat "back catalog" list at the top of `/archive` (`app/archive/page.tsx`) with a big, premium, interactive month-grid calendar. Each date cell shows a small badge with how many stories were published that day. Clicking a date expands an inline, dense, Twitter/X-style feed of that day's stories (headline + one-line teaser + meta row, no page reload). Clicking any story in that feed goes to the existing `/story/[slug]` full-article page unchanged. The current plain month-by-month link list stays below the calendar as a lightweight fallback/SEO surface — nothing is deleted, only added above it.

This is a **UI/presentation-only** feature. `getDataSource().getArchive()` (see `lib/data/index.ts` / `lib/data/mock.ts`) already returns `ArchiveMonth[]` → `{ month, label, editions: Edition[] }` → `Edition { date, stories: PublishedStory[] }`, which is everything needed (date → story count, date → stories). No data-layer or type changes.
</objective>

  <name>Task: Build CalendarBrowser client component</name>

  <action>
Create a new client component (`"use client"`) `components/CalendarBrowser.tsx`:

Props: `{ archive: ArchiveMonth[] }` (the exact return type of `getDataSource().getArchive()`, imported from `@/lib/types`).

Internal data prep:
- Flatten `archive` into a `Map<string, Edition>` keyed by ISO date, and a sorted list of all months that actually have editions (for clamping navigation).
- Default `viewMonth` state = the month containing the most recent edition date (i.e. `archive[0]`, since archive is newest-first per the existing archive page).
- Default `selectedDate` state = the single most recent edition date, so the feed panel is populated on first paint instead of empty.

Layout, top to bottom:
1. **Month header**: big centered "{Month} {Year}" label (reuse the `label` field already computed by `lib/data`) with prev/next chevron buttons (same visual language as the prev/next edition buttons in `components/FeedView.tsx` — `btn-press`, rounded-full, border-border-app). Disable/hide prev beyond the oldest month with an edition, next beyond the newest.
2. **Weekday row**: Sun–Sat abbreviated labels, small `font-mono uppercase tracking-[0.14em] text-ink-3` labels, 7-column grid.
3. **Date grid**: 7-column grid, standard month-grid layout (leading/trailing blank cells to align the 1st of the month to its weekday, computed from UTC dates — mirror the `new Date(\`${date}T00:00:00Z\`)` / `getUTCDate()` / `timeZone: "UTC"` pattern already used in `app/archive/page.tsx` so this stays timezone-consistent with the rest of the app). Cells are large (e.g. `aspect-square`, rounded-2xl, min touch target ~44px, bigger on desktop) — this must read as "big calendar", not a compact widget.
   - Cell with a published edition: interactive `<button>`, brand-tinted background (`bg-brand-soft` / border `border-brand-border` when not selected), the day number, and a small circular count badge in the corner (e.g. `absolute -top-1.5 -right-1.5 flex size-5 items-center justify-center rounded-full bg-brand text-[10px] font-bold text-[var(--on-accent,#fff)]`) showing `edition.stories.length`. Hover: subtle scale/shadow lift (`hover:scale-105 hover:shadow-md transition-transform`). Selected state: `ring-2 ring-brand` plus a filled brand background instead of soft.
   - Cell with no edition: plain day number, muted (`text-ink-3/50`), no background/border, `disabled`, `cursor-default`, no hover effect.
   - Every interactive cell has `aria-label` like `"22 July 2026 — 3 stories"` and `aria-pressed={selectedDate === date}`.
   - Use `whileTap={{ scale: 0.95 }}` (framer-motion `motion.button`) on interactive cells for the "cute/premium" tactile feel.
4. **Selected-day feed panel** (below the grid, only rendered when `selectedDate` is set and has an edition — wrap the mount/unmount in framer-motion `AnimatePresence` with a slide-down + fade transition):
   - Small header row: formatted date (`formatDate(selectedDate)`), story count, and a "View full edition →" link to `/edition/${selectedDate}` (secondary/outline style, does not compete visually with the feed).
   - Twitter/X-style dense vertical list, one row per story, **not** cards — thin `divide-y divide-border-app/70` rows inside a `surface-standard` or `glass-card` container:
     - Each row is a `<Link href={`/story/${story.slug}`}>` (this hits the existing, unmodified `/story/[slug]` page).
     - Row content: category short label as a small colored pill (`CATEGORY_META[story.category].shortLabel`, reuse the same pill styling already used in `StoryCard`'s `rail`/`default` variants for visual consistency), then the story title (`font-semibold`, 1 line, truncate), then a 1-line teaser below it if `story.summary` exists (`text-ink-2 text-[13px] line-clamp-1`), then a meta row (`formatReadingTime(story.readingTimeMin)` · "Must cover" badge only if `story.decision === "must_cover"`).
     - Hover state matches existing row/card interaction patterns (`hover:bg-elevated-muted/70` or similar — check `StoryCard`'s `rail` variant for the exact classes and mirror them).
   - Empty edge case (shouldn't normally happen since only dates with editions are clickable, but defensively): if `selectedDate` has zero stories, show nothing / don't render the panel.
5. If `selectedDate` is set but has no matching edition (defensive), don't render the panel at all.

Keep this component self-contained — no new files beyond this one, no changes to `lib/`.
