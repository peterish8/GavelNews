# Quick Task 260723-l3i: Fix dark-theme color issues (why-it-matters box, TTS accent, footer)

## Task

User flagged (via screenshot of the dark-theme story reader):
1. The "Why it matters for CLAT" callout box (`.surface-emphasis`) renders as a muddy grey-brown in dark mode.
2. The TTS reader-pill's play button / progress bar / language chip render pink instead of a "proper red".
3. The site footer background is not visible / wrong in dark mode.

## Root causes (found by reading `app/tokens.css` + `app/globals.css`)

1. `app/tokens.css`'s `:root.dark` block never defines `--gv-warn-soft` (or several sibling `--gv-*-soft`/`--gv-accent*`/`--gv-gold*` tokens). `.surface-emphasis`'s dark rule (`app/globals.css:313-316`) color-mixes `--gv-warn-soft` into a near-black base — since the var falls through to the **light** theme's pale peach (`#FBEEDA`), the result is a washed-out tan-grey, not a proper dark warm tone.
2. `--brand` in `:root.dark` (`app/tokens.css:260`) is `#F87171` (a light coral/salmon), while `--primary`/`--destructive` dark tokens are proper saturated reds (`#EF4444`/`#DC2626`-family). The TTS pill (`components/StoryReader.tsx`) uses `bg-brand`/`text-brand`/`border-brand` throughout, so it inherits the coral-reads-as-pink color site-wide in dark mode.
3. Both `.surface-muted` (`app/globals.css:276`) and `Footer.tsx`'s inline arbitrary background reference `var(--elevated, #fff)` — `--elevated` is never defined anywhere in `tokens.css` (the real token is `--app-elevated`), so this always falls back to the `#fff` default regardless of theme, producing a whitish band that clashes with dark mode.

## Tasks

1. **Fix `--brand` dark tokens (tokens.css)**
   - Files: `app/tokens.css`
   - Action: In `:root.dark`, change `--brand: #F87171` → `#EF4444`, `--brand-hover: #EF4444` → `#DC2626`, `--brand-border: rgba(248, 113, 113, 0.28)` → `rgba(239, 68, 68, 0.28)` (leave `--brand-soft` — already red-based).
   - Verify: dark-mode brand color used by TTS pill/links now resolves to a saturated red, not coral/pink.
   - Done: `--brand` family in `:root.dark` uses red hex values consistent with `--primary`/`--destructive`.

2. **Add missing dark-mode Gavelogy extended-palette tokens (tokens.css)**
   - Files: `app/tokens.css`
   - Action: In `:root.dark`'s "Gavelogy status" block, add the missing `-soft`/accent/gold tokens that currently silently fall back to light-theme pastel values: `--gv-warn-soft`, `--gv-success-soft`, `--gv-danger-soft`, `--gv-accent`/`-hover`/`-soft`/`-border`, `--gv-gold`/`-soft`/`-border`.
   - Verify: `.surface-emphasis` in dark mode now mixes a proper dark amber-brown instead of a pale peach; other `-soft` badge/button surfaces (FavoriteButton, CompleteButton, NavSearch, SearchResults) get consistent low-opacity dark overlays instead of inherited light pastels.
   - Done: every extended-palette token used by `--color-*-soft` Tailwind mappings has an explicit dark value.

3. **Fix the `--elevated` → `--app-elevated` typo (globals.css + Footer.tsx)**
   - Files: `app/globals.css`, `components/Footer.tsx`
   - Action: Replace the two `var(--elevated, #fff)` references with `var(--app-elevated)` (the token that actually exists and has correct light/dark values).
   - Verify: Footer background now tracks the dark-mode elevated surface color instead of always resolving to white; `.surface-muted` consumers (FeedView, StoryCard, EmptyState) get the same fix for free.
   - Done: no remaining references to the undefined `--elevated` token.

## Notes

These are genuine bugs (undefined var, missing dark overrides) rather than a style preference, and the file's own header says "All --variables live here" — there is no other appropriate place to fix them in this repo. Flag to the user that if there's an upstream shared design-token source, this fix should be mirrored there too.
